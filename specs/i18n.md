# Internationalization (i18n) — French / English

## Status
Draft

## Context
The UI is currently English-only (hard-coded strings in pages and components). The product
is used in French; users should be able to read the app in French and switch language from
the interface. This is a cross-cutting front-end feature touching every component that renders
text. It is independent from the data (person names, places, notes stay as entered by the user
— only the UI chrome is translated).

## Goal
1. All user-facing UI strings come from translation resources, not literals.
2. French and English are both available; French is the default.
3. A visible language switcher (flag/locale icons) toggles the language at runtime.
4. The chosen language persists across reloads.

## Functional rules
- Supported locales: `fr` (default) and `en`.
- Initial locale resolution order: stored preference (localStorage) → browser language → `fr`.
- Switching language updates the whole UI immediately, no reload.
- Date/number formatting follows the active locale (e.g. `Intl.DateTimeFormat`).
- User-entered content (person fields, tree titles, notes) is never translated.
- Validation/error messages shown to the user are localized; server `code` values stay stable
  and are mapped to localized messages on the client.

## Architecture (proposed)

### Client
- Library: `react-i18next` + `i18next` (de-facto standard, lazy namespaces, pluralization).
- `src/i18n/index.ts` — i18next init (resources, default/fallback locale, detection).
- `src/i18n/locales/fr.json`, `src/i18n/locales/en.json` — translation resources, namespaced
  (`common`, `home`, `tree`, `person`, `errors`).
- `src/i18n/useLocale.ts` — hook exposing current locale + setter that writes localStorage.
- `src/components/LanguageSwitcher.tsx` — icon buttons (🇫🇷 / 🇬🇧 or `FR`/`EN`) in the home header
  and the tree toolbar; calls the locale setter.
- Replace all literal strings in `pages/` and `components/` with `t('namespace.key')`.
- Map `ApiError.body.code` → localized message via the `errors` namespace.
- `main.tsx` — import the i18n init before rendering.

### Server
- No change required (UI-only). The server keeps returning stable `code` fields for business
  errors so the client can localize them.

## Data Model
None. No persistence beyond a `localStorage` key (e.g. `ancestry.locale`).

## UI / UX
- Language switcher: two compact icon buttons (or a small dropdown) showing the available
  locales, active one highlighted. Placed in the home header and the tree toolbar.
- Default to French on first visit.

## Test Plan
- Unit: i18n init returns FR strings by default; switching to EN returns EN strings.
- Component: `LanguageSwitcher` toggles locale and persists it; a sample page renders the
  translated heading after a switch.
- Manual: switch language on home and tree pages; reload → language preserved; verify dates
  format per locale.

## Out of Scope
- Locales beyond FR/EN.
- Translating user content or server-stored data.
- RTL layouts.
- Server-side localization / Accept-Language negotiation.

## Open Questions
- Switcher style: flag icons vs `FR`/`EN` text toggle (decide at planning time).
- Whether to extract a shared `packages/shared` for error-code↔message mapping (probably not yet).
